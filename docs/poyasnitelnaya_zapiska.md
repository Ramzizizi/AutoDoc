## 1. Введение и постановка задачи

### 1.1. Цель проекта

Разработка веб-системы «База правовой аналитики» — специализированной базы знаний для юристов и правовых аналитиков организации. Система обеспечивает структурированное хранение правовых материалов, их связывание и эффективный поиск.

### 1.2. Актуальность

Современный юрист при подготовке правового заключения вынужден:
- вручную искать нормы в системах КонсультантПлюс / Гарант;
- отдельно отслеживать судебную практику по каждому вопросу;
- вручную агрегировать материалы в Word-документ;
- каждый раз начинать поиск заново, не сохраняя результаты.

По данным опроса практикующих юристов, подготовка одного правового заключения занимает в среднем **4–6 часов**, из которых 60–70% уходит на поиск и систематизацию источников.

### 1.3. Задачи проекта

1. Создать структурированное хранилище нормативных правовых актов, судебных решений и правовых заключений с поддержкой тегирования и связей между документами.
2. Реализовать полнотекстовый поиск с учётом морфологии русского языка.
3. Добавить фасетную фильтрацию (по типу, отрасли, тегам, дате).
4. Визуализировать связи между документами в виде графа.
5. Автоматизировать сборку правового заключения и экспорт в .docx.
6. Предоставить аналитический дашборд по накопленной базе.
7. Разграничить доступ по ролям (Редактор / Аналитик / Читатель).

### 1.4. Ограничения проекта

- Система предназначена для внутреннего использования в организации, не является публичным ресурсом.
- Система не содержит нейросетевых компонентов и ИИ-резюмирования.
- Юридическая достоверность загруженных материалов — ответственность редакторов базы.

## 2. Анализ предметной области

### 2.1. Сущности предметной области

**Норма права (НПА)** — первичный правовой документ: статья закона, кодекса, указа, постановления или приказа. Характеризуется видом документа, номером статьи, текстом, датой вступления в силу и отраслью права.

**Судебная практика** — решение суда по конкретному делу. Содержит номер дела, суд, дату решения, тезис (краткое содержание), полный текст и связи с нормами, которые суд применил.

**Правовое заключение** — аналитический документ, подготовленный юристом организации: постановка вопроса, анализ применимых норм и практики, вывод. Связан с использованными нормами и делами.

**Отрасль права** — классификатор: Гражданское, Трудовое, Административное, Семейное, Уголовное и т.д. Используется для фасетной фильтрации.

**Теги** — свободные метки для дополнительной классификации (договор, неустойка, алименты и т.п.).

### 2.2. Связи между сущностями

```
ОтрасльПрава (1) ──────── (N) Норма
ОтрасльПрава (1) ──────── (N) СудебнаяПрактика

Норма         (M) ──────── (N) СудебнаяПрактика
Норма         (M) ──────── (N) Заключение
СудебнаяПрактика (M) ─── (N) Заключение
```

Все связи «многие-ко-многим» (ManyToMany) — одна норма может применяться в нескольких делах, одно дело может опираться на несколько норм.

### 2.3. Процесс подготовки правового заключения (TO-BE)

1. Юрист вводит запрос в поисковую строку.
2. Система возвращает релевантные нормы и практику с ранжированием.
3. Юрист фильтрует результаты по отрасли, тегам, дате.
4. Юрист добавляет нужные материалы в корзину конструктора.
5. Юрист вводит текст правового вопроса и вывода.
6. Система генерирует готовый .docx-документ.
7. Заключение сохраняется в базе со связями на источники.

## 3. Архитектура системы

### 3.1. Общая архитектура

Система построена по классической трёхзвенной архитектуре:

```
Браузер (Bootstrap 5 + HTMX)
        ↕  HTTP / WebSocket
Веб-сервер Gunicorn → Django 5.1
        ↕  Django ORM + psycopg2
PostgreSQL 16
```

Все компоненты запускаются в Docker Compose.

apps/

## 5. Реализованные модули

### 5.1. Полнотекстовый поиск

Реализован в `apps/search/views.py`. Ключевые технические решения:

**Для PostgreSQL** (production):
```python
sq = SearchQuery(query, config='russian', search_type='websearch')
vec = SearchVector('title', 'article', 'text', config='russian')
Norm.objects.annotate(
    search=vec,
    rank=SearchRank(vec, sq),
    snippet=SearchHeadline('text', sq, config='russian',
                           max_words=40, min_words=20)
).filter(search=sq).order_by('-rank')
```

**Для SQLite** (разработка — fallback):
```python
Norm.objects.filter(
    Q(title__icontains=query) | Q(text__icontains=query) | Q(article__icontains=query)
)
```

Параметр `search_type='websearch'` поддерживает:
- `договор аренды` — оба слова с учётом стемминга
- `"надлежащим образом"` — точная фраза
- `-алименты` — исключение слова

Результаты поиска объединяются из трёх моделей (нормы, практика, заключения), сортируются по убыванию ранга (`.sort(key=lambda x: x['rank'], reverse=True)`).

**HTMX:** При наличии заголовка `HX-Request` вью возвращает только частичный шаблон `results_partial.html`, без перезагрузки страницы.

### 5.2. Фасетный поиск

Боковая панель с четырьмя группами фильтров:

1. **Тип материала** — нормы / практика / заключения
2. **Отрасль права** — со счётчиками (`_get_branch_facets`, агрегация по GROUP BY)
3. **Теги** — топ-15 тегов в текущих результатах
4. **Диапазон дат** — `date_from` / `date_to`

Все фильтры комбинируются, каждый клик обновляет результаты через HTMX.

### 5.3. Карточки и граф связей

Каждая сущность имеет детальную страницу с:
- метаданными и тегами (кликабельными ссылками на поиск);
- блоком «Связанные материалы» (до 8 элементов каждого типа, без N+1 — все связи вычисляются одним запросом в вью);
- интерактивным графом связей.

**Граф** строится через JSON API (`/knowledge/graph/<type>/<pk>/`), который возвращает узлы и рёбра:

```json
{
  "nodes": [
    {"id": "norm-1", "label": "ст. 309 ГК РФ", "group": "center", "url": "/..."},
    {"id": "case-5", "label": "А53-12345/2022", "group": "case",   "url": "/..."}
  ],
  "edges": [
    {"from": "norm-1", "to": "case-5"}
  ]
}
```

Визуализация на `vis-network`: клик по узлу открывает соответствующую карточку.

### 5.4. Конструктор заключения

**Корзина** хранится в сессии пользователя (`request.session['basket']`):
```python
{'norms': [1, 5, 12], 'cases': [3, 7]}
```

**Операции с корзиной:**
- `POST /builder/basket/add/<type>/<pk>/` — добавить (идемпотентно, дублей нет)
- `POST /builder/basket/remove/<type>/<pk>/` — удалить (HTMX возвращает обновлённый partial)
- `POST /builder/basket/clear/` — очистить

**Экспорт .docx** (`POST /builder/export/`):
- Документ формируется через `python-docx` с отступами ГОСТ
- Разделы: заголовок → дата/автор → правовой вопрос → нормы → практика → заключение
- Имя файла кодируется по RFC 5987 для корректной передачи кириллицы:
  ```
  Content-Disposition: attachment; filename="zaklyuchenie.docx"; filename*=UTF-8''...
  ```

**Защита:** все операции корзины доступны только ролям `Аналитик` и `Редактор` (`@analyst_required`). Все мутирующие endpoints защищены `@require_POST` и CSRF.

### 5.5. Дашборд аналитики

Страница `/analytics/` (доступна Аналитику и Редактору) содержит 4 диаграммы Chart.js и 4 KPI-счётчика.

Данные передаются как JSON-строки в контексте шаблона:

```python
norms_by_branch_labels = json.dumps([...], ensure_ascii=False)
norms_by_branch_data   = json.dumps([...])
```

| Диаграмма | Тип | Запрос |
|---|---|---|
| Нормы по отраслям | Столбчатая | `Norm.objects.values('branch__name').annotate(Count)` |
| Нормы по видам НПА | Круговая | `Norm.objects.values('norm_type').annotate(Count)` |
| Практика по отраслям | Горизонтальная | `CourtCase.objects.values('branch__name').annotate(Count)` |
| Динамика по годам | Линейная | `CourtCase.objects.annotate(year=TruncYear('decision_date'))` |

## 6. Безопасность и разграничение доступа

### 6.1. Роли

Реализованы через Django Groups. Три группы создаются автоматически при `migrate` (через сигнал `post_migrate`):

| Роль | Что может |
|---|---|
| Редактор базы | Всё: поиск, конструктор, дашборд, admin |
| Аналитик | Поиск, конструктор, дашборд, только публичные и свои заключения |
| Читатель | Поиск и просмотр, только публичные заключения |

Иерархия: Редактор ⊃ Аналитик ⊃ Читатель.

### 6.2. Декораторы

```python
@login_required        # стандартный Django
@analyst_required      # кастомный: is_analyst() or 403
@editor_required       # кастомный: is_editor() or 403
```

### 6.3. Защита от уязвимостей

| Уязвимость | Меры защиты |
|---|---|
| CSRF | `{% csrf_token %}` во всех формах; `CsrfViewMiddleware` |
| Open redirect | `url_has_allowed_host_and_scheme()` для параметра `next` |
| SQL injection | Только Django ORM — параметризованные запросы |
| XSS | Автоэкранирование шаблонов Django; `\|safe` только для FTS SearchHeadline |
| Brute force | Стандартные валидаторы паролей Django |
| Информационное раскрытие | `DEBUG=False` в production; кастомный `403.html` |

## 8. Инструкция по развёртыванию

### 8.1. Требования к серверу

- Docker Engine 24+ и Docker Compose V2
- ОС: Linux (Ubuntu 22.04 / Debian 12) или macOS
- RAM: минимум 1 ГБ, рекомендуется 2 ГБ
- Диск: 2 ГБ свободного места

### 8.2. Порядок развёртывания

```bash
# 1. Клонировать репозиторий
git clone <repo-url> && cd legal-analytics

# 2. Настроить окружение
cp .env.example .env
nano .env    # установить SECRET_KEY, пароли

# 3. Собрать и запустить
docker compose up --build -d

# 4. Проверить
docker compose logs web
curl http://localhost:8000/  # должен вернуть 302 → /accounts/login/
```

### 8.3. Обновление

```bash
git pull
docker compose up --build -d
```

Данные в PostgreSQL сохраняются в named volume `postgres_data` между перезапусками.

### 8.4. Пересоздание данных

```bash
docker compose down -v           # удалить тома (данные будут потеряны)
docker compose up --build        # пересоздать с чистой базой
```

## 9. Расчёт экономической эффективности (ROI)

### 9.1. Исходные данные

| Параметр | Значение | Источник |
|---|---|---|
| Среднее время подготовки заключения (AS-IS) | 5 часов | Опрос юристов |
| Среднее время подготовки заключения (TO-BE) | 1,5 часа | Экспертная оценка |
| Количество заключений в месяц на команду 3 юриста | 30 | Типовая нагрузка |
| Стоимость часа работы юриста | 1 000 ₽ | Средняя по РФ |
| Стоимость разработки (15 нед × 5 чел × 40 ч × 500 ₽) | 150 000 ₽ | Оценка команды |

### 9.2. Расчёт

**Экономия времени на одно заключение:**
```
ΔT = 5 ч − 1,5 ч = 3,5 ч
```

**Экономия в месяц:**
```
E_мес = ΔT × 30 заключений × 1 000 ₽/ч = 105 000 ₽
```

**Срок окупаемости:**
```
T_окуп = 150 000 ₽ / 105 000 ₽ ≈ 1,4 месяца
```

**ROI за год:**
```
ROI = (E_год − C_разр) / C_разр × 100%
    = (1 260 000 − 150 000) / 150 000 × 100%
    = 740%
```

### 9.3. Дополнительные эффекты (не учтены в расчёте)

- Снижение риска пропустить применимую норму или релевантную практику
- Накопление корпоративной базы знаний — не теряется при смене сотрудника
- Единый формат заключений вместо произвольных Word-файлов
